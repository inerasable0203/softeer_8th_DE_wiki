# W3M4 팀 활동 보고서

## 1. 활동 주제

### 팀 활동 요구사항

predefined keywords가 아닌 다른 방법으로 데이터를 나누고자 한다면 어떤 방법이 있는지 팀과 함께 논의하고, 다른 방법을 시도한다.

이번 활동에서는 기존의 단순 키워드 기반 감성 분류 방식의 한계를 확인하고, 이를 보완할 수 있는 여러 대안을 비교하였다.

## 2. 기존 키워드 기반 분류 결과

팀원들은 서로 다른 긍정·부정 단어 목록을 사용하여 Sentiment140 데이터 1,600,000건을 분류하였다.

| 팀원 | Negative | Neutral | Positive | Total |
|---|---:|---:|---:|---:|
| 종민 | 376,052 (23.50%) | 678,017 (42.38%) | 545,931 (34.12%) | 1,600,000 (100%) |
| 용진(Albert) | 89,534 (5.60%) | 1,248,250 (78.02%) | 262,216 (16.39%) | 1,600,000 (100%) |
| 최지욱 | 227,414 (14.21%) | 1,016,801 (63.55%) | 355,785 (22.24%) | 1,600,000 (100%) |

같은 데이터셋을 사용했음에도 단어 목록과 분류 규칙에 따라 결과 차이가 크게 나타났다. 이를 통해 predefined keywords 방식은 구현이 단순하지만, 사용자가 어떤 단어를 선택하느냐에 따라 결과가 크게 달라질 수 있음을 확인하였다.

## 3. Predefined Keywords 방식의 장점과 한계

### 장점

Predefined keywords 방식은 각 트윗에 특정 단어가 포함되어 있는지만 확인하면 되기 때문에 처리 로직이 단순하다. 각 트윗을 서로 비교하지 않고 독립적으로 처리할 수 있어 MapReduce 구조와 잘 맞는다.

```
대규모 데이터
→ Mapper가 각 데이터를 독립적으로 분류
→ 같은 Key끼리 Shuffle
→ Reducer가 감성별 개수 집계
```

예를 들어 Mapper가 다음과 같이 출력하면,

```
positive    1
negative    1
positive    1
```

Reducer는 같은 Key의 값을 합산한다.

```
positive    2
negative    1
```

이처럼 단순하고 독립적인 분류 로직을 수백만 건 이상의 데이터에 병렬 적용할 수 있다는 점이 MapReduce의 핵심 장점이다.

### 한계

Predefined keywords 방식은 사람이 미리 정한 단어에 의존하기 때문에 다음 문제가 발생한다.

* 사전에 없는 표현을 분류하지 못한다.
* love, loved, loving처럼 형태가 다른 단어를 놓칠 수 있다.
* not good처럼 부정어가 포함된 표현을 잘못 분류할 수 있다.
* 하나의 단어만 확인하므로 문장 전체의 감성 강도를 충분히 반영하기 어렵다.
* 팀원마다 단어 목록이 달라지면 결과의 일관성이 낮아진다.

예를 들어 다음 문장은 부정적인 의미이지만 명확한 부정 키워드가 없을 수 있다.

> I expected the car to look much better.

## 5. 팀에서 선택한 방법: VADER

우리 팀은 predefined keywords 방식의 단순함을 유지하면서도 단어의 감성 강도와 부정 표현을 반영하기 위해 VADER를 적용하였다.

### 적용 방법

* `vaderSentiment` 라이브러리의 `SentimentIntensityAnalyzer` 사용
* 트윗별 감성 점수 계산 (-4~+4)
* 최종 점수를 기준으로 Negative, Neutral, Positive 분류
* Mapper에서 감성 Key와 1 출력
* Reducer에서 감성별 개수 합산

### 처리 흐름

```
트윗 입력
→ 텍스트 정규화
→ 단어별 감성 점수 추출
→ 단어별 감성 점수에 전후 문맥 반영
→ 감성 점수 합산 후 느낌표/물음표 반영
→ 감성 점수 정규화
→ 감성 Label 결정
→ Mapper가 Label과 1 출력
→ Reducer가 Label별 개수 집계
```

### 실행 결과

| 감성 | 트윗 수 | 비율 |
|---|---:|---:|
| Negative | 417,690 | 26.11% |
| Neutral | 432,034 | 27.00% |
| Positive | 750,276 | 46.89% |
| 합계 | 1,600,000 | 100.00% |

VADER를 적용한 결과, 단순 키워드 방식보다 Neutral 비율이 줄고 Positive와 Negative 분류가 증가하였다. 이는 단어 포함 여부만 확인하는 것이 아니라 여러 단어의 감성 강도와 부정어, 강조 표현 등을 함께 계산했기 때문으로 볼 수 있다.

다만 이 결과가 실제 정답에 더 가깝다는 것을 의미하지는 않는다. VADER도 사람이 구축한 감성 사전에 기반한 규칙형 분석 도구이므로, 사전에 없는 신조어·도메인 전문 용어·복잡한 문맥을 충분히 해석하지 못할 수 있다. 또한 반어법, 풍자, 문장 간 맥락, 동일 단어의 상황별 의미 차이를 판단하는 데 한계가 있다.

## 6. Vader의 주요 동작 원리

### 텍스트 정규화

1. **이모지 → 텍스트 설명으로 변환**

`polarity_scores()` 맨 처음 단계에서,`:blush:` 같은 이모지를 사전에 등록된 문구(예: `"smiling face"`)로 바꿔치기합니다.

1. **단어 앞뒤에 붙은 구두점 제거**
2. **토큰화**

공백 기준 `split` 적용

### 단어별 감성 점수 추출

분석 대상 단어를 사전(lexicon)에서 찾아 기본 점수(-4~+4)를 가져옵니다.

### 단어별 감성 점수에 전후 문맥 반영

* 부정어(not, never 등)가 앞에 있으면 → 점수에 `N_SCALAR`(음수 배율)를 곱해서 뒤집음
* 정도부사(very, kind of 등)가 있으면 → 점수를 키우거나 줄임
* 그 단어가 대문자이고 문장 안 다른 단어들은 소문자면 → 점수를 더 키움(`C_INCR`만큼)
* but이 문장에 있으면 → but 앞의 감성은 절반(×0.5)으로 낮추고, 뒤는 1.5배로 키움(뒷부분이 더 중요하다는 뜻)

### 감성 점수 합산 후 느낌표/물음표 반영

단어별 점수를 전부 더하고, 느낌표/물음표로 한 번 더 보정한다.

### 감성 점수 정규화

점수를 -1~+1 범위로 정규화한다.

## 7. VADER의 한계

VADER는 predefined keywords 방식보다 다양한 언어 표현을 반영할 수 있지만, 완전한 문맥 기반 모델은 아니다. 주요 한계는 다음과 같다.

* 감성 사전에 등록된 단어와 점수에 의존한다.
* 자동차와 같은 특정 도메인의 전문 용어나 신조어를 잘못 판단할 수 있다.
* 반어법과 풍자를 정확하게 이해하기 어렵다.
* 긴 문장에서 앞뒤 문맥이나 문장 간 관계를 충분히 반영하지 못한다.
* 같은 단어가 상황에 따라 다른 의미로 사용되는 경우를 구분하기 어렵다.
* 영어 중심으로 설계되어 다른 언어나 혼합 언어 데이터에는 추가 전처리가 필요하다.
* 감성 점수 임계값에 따라 Positive, Neutral, Negative 비율이 달라질 수 있다.

예를 들어 다음 문장은 표면적으로 긍정 단어를 포함하지만 실제로는 부정적인 반어 표현이다.

> Great, another car design that looks exactly the same.

따라서 VADER는 빠르고 가벼우며 각 데이터를 독립적으로 처리할 수 있다는 장점이 있지만, **결과의 정확성을 확인하려면 원본 레이블 또는 사람이 검토한 표본과 비교하는 검증 과정이 필요하다.**

## 8. MapReduce를 사용하는 이유

MapReduce를 사용하는 목적은 복잡한 모델을 만드는 데 있지 않다. 핵심은 각 데이터에 독립적으로 적용할 수 있는 비교적 단순한 로직을 여러 노드에서 병렬 실행하고, 결과를 Key 기준으로 쉽게 분류하고 집계하는 데 있다.

이번 VADER 방식은 각 트윗을 독립적으로 분석할 수 있다. 한 트윗의 감성을 계산하기 위해 다른 트윗의 결과가 필요하지 않으므로 Mapper들이 데이터를 나누어 동시에 처리할 수 있다.

```
트윗 1 → 감성 계산 → positive, 1
트윗 2 → 감성 계산 → negative, 1
트윗 3 → 감성 계산 → neutral, 1
```

이후 Shuffle 단계에서 같은 감성 Key가 모이고, Reducer는 개수만 합산한다. 따라서 VADER는 predefined keywords보다 정교하지만, 여전히 MapReduce의 단순하고 독립적인 처리 구조를 유지한다.

## 9. 팀 논의 결과

우리 팀은 predefined keywords 방식을 완전히 복잡한 머신러닝 구조로 대체하기보다, MapReduce의 장점을 유지하면서 분류 규칙을 확장하는 방향을 선택하였다.

최종적으로 다음 요소를 결합하였다.

```
텍스트 정규화
+ 단어별 감성 점수
+ 부정어 및 강조 표현 처리
+ MapReduce 병렬 분류 및 집계
```

이 방식은 단순 키워드 매칭보다 다양한 표현을 처리하면서도 각 트윗을 독립적으로 분석할 수 있다. 따라서 대규모 데이터에 동일한 로직을 병렬 적용한다는 MapReduce의 목적을 유지할 수 있다.

## 10. 결론

Predefined keywords 방식은 구현이 단순하고 대규모 데이터에 빠르게 적용할 수 있지만, 단어 목록과 단순 포함 여부에 지나치게 의존한다는 한계가 있다.

우리 팀은 이를 보완하기 위해 VADER를 사용하였다. VADER는 단어별 감성 강도, 부정어, 강조 표현, 이모티콘 등을 반영하면서도 각 트윗을 독립적으로 처리할 수 있기 때문에 MapReduce 구조에 적합하다. 다만 감성 사전과 규칙에 의존하므로 반어법, 도메인 용어, 복잡한 문맥을 정확히 판단하지 못할 수 있으며, 결과 해석 시 이러한 한계를 함께 고려해야 한다.

**이번 활동을 통해 중요한 것은 항상 더 복잡한 기술을 선택하는 것이 아니라, 처리하려는 문제의 구조에 맞는 방법을 선택하는 것임을 확인하였다.**

* 단순하고 독립적인 대규모 분류·집계에는 **MapReduce**가 적합하다.
* 반복 학습과 여러 단계의 복잡한 처리에는 **Spark**가 적합하다.
* 이번 과제에서는 **VADER와 MapReduce의 조합**이 predefined keywords 방식의 한계를 보완하면서도 대규모 병렬 처리의 장점을 유지하는 적절한 선택이었다.
